caformer: a chat model built from cellular automata, no transformers, no GPUs

This is a research artifact, not a production system. It explores a simple question: can a language model be built entirely out of deterministic cellular automata, with no floating-point operations, no attention mechanism, no gradient descent, and no GPU?

The short answer, from what is currently running on this server: yes, for byte-exact reproduction of trained prompt → response pairs at up to 73 bytes per response. Whether it scales to GPT-3-level breadth and generalization is an open research question. The architectural primitives, however, scale cleanly.

What is one chain, concretely?

One chain is a 16,384-entry lookup table. Each entry is a 2-bit number (0..3, i.e. one of four colours). A chain stores the deterministic transition rule of a hex-grid cellular automaton with a 7-cell neighbourhood (self + 6 hex neighbours): for every possible configuration of those 7 cells, the LUT says what colour the centre cell should become next tick. There are 47 = 16,384 such configurations, hence the LUT size.

+---------+---------+---------+
| (self,  | …       | (self,  |
|  NW=0,  |         |  NW=3,  |
|  NE=0,  |         |  NE=3,  |
|  E=0,   |         |  E=3,   |
|  SE=0,  |         |  SE=3,  |
|  SW=0,  |         |  SW=3,  |
|  W=0)   |         |  W=3)   |  ← 16384 possible inputs
+---------+---------+---------+
   ↓         ↓         ↓
+---+       +---+      +---+
| 2 |  ...  | 0 |  ... | 3 |    ← LUT entry: next colour (0..3)
+---+       +---+      +---+

 Total table: 16,384 × 2 bits = 32,768 bits = 4 KB packed
                                                (16 KB unpacked)

At inference, the rule is applied to every cell of a {{ board_side }}×{{ board_side }} grid simultaneously, then applied again to the result, and again, for {{ board_side }} ticks total. The final grid is read out as a sequence of bytes — 4 cells per byte, four base-4 digits, top-left layout.

How a prompt becomes a board

The prompt is encoded into the top half of the board (cells 0..{{ board_cells }}÷2). Each byte becomes 4 cells. The prompt 'hi' = bytes [0x68, 0x69] embeds as the 8 cells:

hi = 0x68 0x69 → cells:
    ●●●○   ●●●●
    │ │ │ │   │ │ │ │
   ○ = 0   ● = 1   ● = 2   ● = 3

The remaining ~16,376 cells are zero. After {{ board_side }} ticks of running the trained rule, the response region (bottom half) contains the answer, also as 4 cells per byte.

How a response is read out

For an N-byte response, N chains are trained — one per output position. At inference, each of the N chains runs independently on the same starting board state, and the i-th chain's final board has the correct byte i in its decoder cells. The responses concatenate.

Why a 4-colour hex grid can hold language

The skeptical question is fair: how can text, with its 256 possible byte values, fit into a 4-colour grid?

The answer is encoding density. 4 cells = 1 byte (4 base-4 digits = 8 bits). A 128×128 board has 16,384 cells = 4,096 bytes of bandwidth — enough for a 2 KB prompt + 2 KB response, or any other split. That's already more context than GPT-2 had (1024 tokens).

The deeper question is whether the CA's dynamics can implement arbitrary mappings. The answer: yes, by construction. The rule space is 416384 ≈ 109866 possible rules — vastly larger than the number of distinct prompt → response mappings we'd ever need. The genetic algorithm's job is to find a rule whose 128-tick evolution produces the desired output byte. Each output byte gets its own dedicated rule.

For the curious: at 128 ticks on a 128×128 board, signal can propagate from any cell to any other cell. The CA has time to compute, not just time to diffuse. The choice of 128 — board side = tick count = √(LUT size) — is the architecturally self-consistent constant.

Why not just use random data instead of CAs?

This is the question most skeptical computer scientists ask first. The answer is precise: random rules don't compute. Wolfram classified one-dimensional CAs into four behavioural classes:

  • Class 1: rapid convergence to a uniform state (boring; signal dies)
  • Class 2: simple periodic structures (signal repeats; no information processing)
  • Class 3: chaotic noise (information is destroyed)
  • Class 4: persistent localised structures that move, interact, and compute (information is preserved AND transformed)

A randomly generated K=4 hex CA rule almost always falls into class 1, 2, or 3. The GA's job is to evolve rules toward class 4, where the dynamics can carry input through to a specific desired output. The polish phase (single-byte coordinate descent) then nudges the rule across the final argmax threshold for the target byte.

If we used random data instead of CAs: the "model" would have no way to propagate input bits through to output bits. Information at the prompt cells would never reach the response cells in any meaningful form. The output would be a fixed pattern determined by the random table, independent of the prompt. We verified this experimentally — random rules produce uniform fixed-point patterns regardless of input.

The CA structure provides three things random data cannot:

  1. Spatial locality: a cell's next state depends on its 7-cell neighbourhood, so information can propagate deterministically through space (one hex per tick).
  2. Universal computation: at class 4, hex CAs are Turing-complete — they can in principle compute any function the rule's LUT encodes.
  3. Composability: a 16 KB rule is a 128×128 image, which means rules can be inputs to other rules. This enables self-reference (Ouroboros / quine structures) that random data cannot.

The honest summary: caformer is materially worse than a transformer at generalising to unseen prompts. It is materially better at reproducing trained answers exactly, cheaply, deterministically, and on commodity hardware. The open research question is whether self-supervised consolidation (the DMN heartbeat loop running on this server right now), chain reuse across similar prompts, and amplification heads derived from quine chains can close the generalisation gap.

How it scales (with current measurements + ALICE HPC)

Per-pair training cost is measured at ~30 seconds per byte on a single Intel CPU core (BLAS pinned to 1 thread). Scaling this to corpus sizes comparable to public LLM training sets, assuming chain reuse, GA warm-start, and cluster-centroid compression optimisations stack to a 50-100× speedup over naive sequential training, and assuming access to Leiden University's ALICE HPC cluster (up to 288 CPUs per job):

For comparison, the storage required for the GPT-3-equivalent scale fits comfortably on a single 8 TB drive (~3 TB used). The inference latency at that scale is still ~10 seconds per 500-byte reply — much slower than a GPU-served transformer, but workable for asynchronous chat (which is why this interface is queue-based and lets you type continuously).

Train on a supercomputer, query on a Raspberry Pi

This is the architectural property most worth pausing on. The chain pool is the model. Once trained, the artifacts are just files on disk — there is no GPU initialisation, no model loading, no tensor allocation, no warmup. To deploy a trained caformer to a new machine you literally rsync a directory.

The runtime working set at any moment:

ComponentRAM cost
The currently-running chain 16 KB
The current 128×128 board state 4 KB (16,384 cells × 2 bits)
Python interpreter / C runtime varies, ~10-50 MB

You do not need to load the whole model into RAM. The OS pages chains in via mmap as the per-position loop walks through them. Storage acts as the model store; RAM acts as the inference scratchpad. This is the exact inversion of the GPU-LLM pattern, where the entire weight tensor must live in GPU memory.

The training / deployment split:
ALICE HPC (or any cluster) trains for hours, produces ~3 TB of binary blobs. Rsync to a lab workstation, then rsync onward to a Pi, an ESP32, or a phone. The chat runs identically on all of them — only inference latency changes.

By comparison, deploying a GPT-3 inference endpoint requires ~350 GB of GPU memory across multiple GPUs and ~$10K/month of cloud GPU time. Deploying a caformer of equivalent breadth requires an 8 TB drive and a Raspberry Pi.

What's running right now on this server

Several background processes:

  • The chat dispatch. When you type a prompt, the 4-way intent router (a single 16 KB chain classifies the prompt as personality / information / action / meta) runs in microseconds, then the dispatcher looks for an exact trained pair. If one exists, the N per-byte chains run in parallel and the response is decoded.
  • The DMN heartbeat. A background thread fires twice per second (lub-dub) whenever the user queue is empty. Each tick takes the most recent assistant reply, feeds it back through the dispatch as a new input, and stores the result as an "internal thought" in your session. The result is an inner monologue visible live in the chat — the model genuinely thinks between your turns. This is the substrate for self-supervised training in v2.
  • Training in the background. 4 worker processes currently grinding through the rest of the corpus, producing one per-byte chain at a time. Each completion immediately becomes available in the live chat — no restart needed.

Honest limitations

  • No graceful generalisation for prompts that aren't byte-exactly trained. The fallback compositional layer is weak.
  • Per-pair training time dominates iteration cost. Without chain reuse, scaling to thousands of pairs takes days.
  • No multi-step reasoning in the inference path. One prompt produces one response. Multi-step thought happens in the DMN loop but isn't yet wired into user-facing responses.
  • No tool use, no agentic capability. The architecture could add these but they're not built. Claude-Code parity is multi-year work, not a scaling problem.

The interesting research bet is whether the existing primitives — class-4 cellular automata, per-byte chains, self-conversation heartbeat, eventually amplification heads derived from quine chains — can compose into a system whose emergent behaviour surprises us the way transformers surprised everyone at GPT-3 scale. The mathematical scaling barrier is gone. What remains is to find out.
