# cellular — ESP32-S3 SuperMini port of the s3lab Cellular sublab

The full Cellular sublab on a stick. 16×16 hex-tiled population of
K=4 hex CA genomes, tournament GA every 500 ms, **ST7735 1.44″
128×128** TFT (`INITR_144GREENTAB`, `setRotation(0)`) showing the
population grid as colored tiles. This is the **recommended** panel
variant — feature-equivalent to the `/s3lab/cellular/tft/` browser
emulator and sized to fit live render mode.

The 80×160 sister variant lives at `../esp32_s3/`.

## Why PSRAM

Per population cell:
  * 4096 B genome (K=4 packed)
  * 4 B palette
  * 256 B grid_a + 256 B grid_b
  * 8 B score + timestamp

Total: ~4620 B per cell × 256 cells = ~1.13 MiB. The supermini has
320 KB SRAM + 8 MB PSRAM. Population goes in PSRAM via
`heap_caps_malloc(MALLOC_CAP_SPIRAM)`. SRAM is reserved for the GA's
working buffers, the WiFi stack, and the score/order arrays
(~2.3 KB) used by Hunt/Refine.

## Provision WiFi (first boot)

Without `/wifi.txt`, the board comes up as `hexca-cellular-setup`
(password `helloboard`, IP 192.168.4.1). From a phone or laptop:

```
curl -X POST http://192.168.4.1/wifi -d 'ssid=YOUR_SSID&password=YOUR_PASS'
```

After it joins, find it at `http://hexca-cellular.local/`.

## Render modes

Two render paths, switchable at runtime:

- **dominant** — each population cell drawn as a single 7×6 hex tile
  coloured by its current dominant palette index. Cheap (~21 KB SPI
  per frame).

- **live** — every cell drawn as a 6×6 subsample of its 16×16
  internal CA. Each rendered pixel = a sample of the actual grid;
  inner hex stagger (odd sub-rows shifted +1 px) and outer hex
  stagger (odd population rows shifted +SUB_W/2) tessellate at two
  scales. Tiles abut without gutters by sharing the overflow column
  with neighbours. ~22 KB SPI per frame.

```
curl -X POST hexca-cellular.local/render-mode -d mode=live
curl -X POST hexca-cellular.local/render-mode -d mode=dominant
```

## Endpoints

| method | path                | what                                    |
|--------|---------------------|-----------------------------------------|
| GET    | /                   | status HTML                             |
| GET    | /info               | JSON status (uptime, heap, PSRAM, rounds, render mode, auto state) |
| POST   | /wifi               | `ssid=…&password=…` — saves + reboots   |
| POST   | /reset              | bootstrap a fresh random population     |
| POST   | /pause / /resume    | freeze / unfreeze the GA loop           |
| POST   | /render-mode        | `mode=dominant\|live`                   |
| POST   | /hunt               | score, sort, refresh bottom 128 cells   |
| POST   | /refine             | clone elite into 255 mutated copies     |
| POST   | /auto               | `mode=on\|off&delay=N` — N seconds, 1..60 |
| POST   | /palette-reroll     | `invent_palette()` per cell             |
| POST   | /palettes           | raw 1024 B body (256 × 4 ANSI indices); browser computes image-derived palettes and pushes |
| GET    | /agent?r=R&c=C      | download an agent's HXC4 genome.bin (4104 B) — round-trips into s3lab/automaton/taxon |

## Cost estimates (timing on 240 MHz Xtensa)

| operation             | wall time |
|-----------------------|-----------|
| tick (256 × step)     | ~5 ms     |
| round (1 × tournament)| ~6 ms     |
| Hunt or Refine        | ~1.0 s    |
| live full repaint     | ~12 ms    |
| dominant diff repaint | ~5 ms     |

A Hunt or Refine call blocks the HTTP server for ~1 s. The auto-
refine loop ticks the same call inside `loop()` between regular
tournament rounds — at the default 3 s cadence the device stays
fully responsive.
